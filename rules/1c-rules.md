## Правила написания кода на 1С

- Избегай комментариев, которые повторяют очевидное. Комментарии допускаются только когда добавляют ценность: мотивация, нетривиальный алгоритм, ограничения, технический долг
- Порядок областей в общем модуле: `ПрограммныйИнтерфейс` (public) → `СлужебныйПрограммныйИнтерфейс` (internal, внутри подсистемы) → `СлужебныеПроцедурыИФункции` (private, внутри модуля)
- Перед написанием кода проверяй в общих модулях и модулях менеджеров экспортные методы, которые можно переиспользовать
- Не используй венгерскую нотацию: `МассивКонтрагентов` — нет, `Контрагенты` — да
- Не используй в именах переменных имена из глобального контекста 1С (Документы, Справочники, Регистры, Метаданные, Константы и др.)
- Для процедур используй глаголы: `ЗаполнитьТаблицу()`. Для функций-предикатов — вопросительную форму: `ЭтоНовый()`, `ЕстьОшибки()`
- Форматирование запросов — текст без отступа на новой строке после `=`:
```bsl
Запрос.Текст =
"ВЫБРАТЬ
|	Таблица.Поле КАК Поле
|ИЗ
|	Таблица";
```
- Для результата запроса используй промежуточную переменную: `РезультатЗапроса = Запрос.Выполнить()`, не `Запрос.Выполнить().Выгрузить()`
- Всегда давай псевдонимы полям через `КАК`: `Контрагенты.ИНН КАК ИНН`
- Перед написанием запроса проверяй реквизиты метаданных (существование, имена, типы)
- Не используй запросы внутри цикла — используй пакетные запросы и временные таблицы
- Не получай реквизиты из ссылки через точку — это загружает весь объект. Используй `ОбщегоНазначения.ЗначениеРеквизитаОбъекта()`, `ЗначенияРеквизитовОбъекта()`, `ЗначениеРеквизитаОбъектов()`, `ЗначенияРеквизитовОбъектов()`
- При повторяющихся вычислениях в цикле используй кеширование через Соответствие
- В модулях форм предпочитай `&НаСервереБезКонтекста` вместо `&НаСервере`. Минимизируй переходы клиент-сервер
- Вместо `ОписаниеОповещения` используй `Асинх`-методы где возможно
- Не используй `Сообщить()`. На сервере: `ОбщегоНазначения.СообщитьПользователю()`, на клиенте: `ОбщегоНазначенияКлиент.СообщитьПользователю()`
- Не глотай исключения молча — логируй через `ЗаписьЖурналаРегистрации`. В `Попытка/Исключение` обрабатывай только ожидаемые исключения
- Для текущей даты на сервере используй `ТекущаяДатаСеанса()`, не `ТекущаяДата()`
- Проверяй код на конкурентность и необходимость блокировок. Учитывай внешние транзакции (например, транзакция записи объекта)
- Ограничение строки — 120 символов (если можно корректно перенести).
- Не используй тернарный оператор `?()`
- После редактирования кода выполняй ревью: стиль, читаемость, корректность, граничные случаи, безопасность, конкурентность

### Запросы (дополнение)
- Не используй подзапросы в ВЫБРАТЬ — это N+1 выполнений. Используй LEFT JOIN с агрегацией
- Фильтруй виртуальные таблицы через параметры, не через ГДЕ — иначе полное сканирование
- Используй ПЕРВЫЕ N когда нужно ограниченное количество записей

### Метрики качества
- Длина метода: <= 200 строк (жёсткий лимит), > 100 строк — кандидат на декомпозицию
- Вложенность: < 5 уровней (жёсткий лимит)
- Параметры функции: <= 5 (жёсткий лимит)

### Именование (дополнение)
- Boolean-переменные — только положительные имена: `ПроверкаПройдена`, не `ПроверкаНеПройдена`
- Магические числа запрещены — выноси в именованные переменные
- Yoda-синтаксис запрещён: `Если Сумма = 0`, не `Если 0 = Сумма`
- Не сравнивай boolean с Истина/Ложь: `Если ЭтоНовый()`, не `Если ЭтоНовый() = Истина`

### Запрещённые конструкции
- Модальные вызовы запрещены (ПоказатьВопрос, ПоказатьПредупреждение, ВвестиЧисло и т.д.)
- Выполнить()/Вычислить() запрещены
- COM-объекты запрещены (без явного указания)
- Hardcoded credentials запрещены

### Архитектура
- Используй Early Return для уменьшения вложенности — ранний выход вместо глубокой вложенности
- Приоритет модификации типового кода: подписки на события > расширения > изменение типового кода
- В расширениях: &Перед/&После (предпочтительно), &Вместо — только с ПродолжитьВызов()
- Операции > 10 секунд — выноси в фоновые задания с индикатором прогресса

### Формы (дополнение)
- 5 обязательных областей в модуле формы (даже если пусты): ОбработчикиСобытийФормы, ОбработчикиСобытийЭлементовШапкиФормы, ОбработчикиСобытийЭлементовТаблицыФормы, ОбработчикиКомандФормы, СлужебныеПроцедурыИФункции
- Модификация типовых форм — только программно через ПриСозданииНаСервере (или подписку/расширение)
- Перед записью — вызывать ПроверитьЗаполнение()

### Транзакции и привилегированный режим
- Транзакции максимально короткие, без пользовательского взаимодействия внутри
- УстановитьПривилегированныйРежим(Истина) — использовать осторожно, всегда выключать после операции

### Коллекции
- ЗаполнитьЗначенияСвойств() для массового присвоения свойств
- НайтиПоЗначению()/Найти() для поиска в коллекциях вместо ручного перебора
- Для больших коллекций — Соответствие для O(1) поиска вместо O(n)

### Документирование
- Документируй все Export процедуры/функции: описание, параметры с типами, возвращаемое значение

### Стиль
- Отступы — TAB, не пробелы
- Одна инструкция на строку
- При работе в проекте с PREFIX — все новые объекты, реквизиты, элементы форм, роли именуются с префиксом
- При модификации типового кода — комментарии модификаций через шаблоны COMMENT_OPEN/COMMENT_CLOSE
- Суффиксы серверных общих модулей: *ВызовСервера, *МодульОбъекта, *МодульМенеджера; клиентских: *Клиент
- Выравнивание `=` в группах присваиваний для улучшения читаемости

## Использование MCP инструментов

### Важные цепочки вызовов
- `its_help` → **обязательно** `fetch_its` — its_help возвращает только ID документов, для чтения полного текста вызывай fetch_its
- `docinfo` — для поиска по точному имени объекта/метода; `docsearch` — для поиска по описанию когда имя неизвестно
- `syntaxcheck` — не более 3 вызовов за цикл работы

### Workflow при написании кода
1. `templatesearch` — найти похожие реализации и паттерны
2. `ssl_search` — найти функции БСП для переиспользования
3. `docinfo`/`docsearch` — проверить встроенные функции платформы
4. ...написать код...
5. `syntaxcheck` — проверить синтаксис
6. `check_1c_code` — найти логические проблемы и проблемы производительности
7. `review_1c_code` — проверить стиль и стандарты

### Workflow при ревью кода
1. `check_1c_code` — техническая проверка
2. `review_1c_code` — стиль и стандарты
3. `its_help` → `fetch_its` — проверить спорные места по стандартам ITS

### Workflow при оптимизации
1. `check_1c_code` — выявить проблемы
2. `rewrite_1c_code` (goal: optimize) — получить оптимизированную версию от AI
3. `its_help` → `fetch_its` — стандарты ITS по производительности

## MCP-сервер rlm-tools-bsl (анализ кодовой базы)

Отдельный MCP-сервер `rlm-tools-bsl` — токен-эффективный анализ BSL-кода без предварительной RAG-индексации. Не заменяет инструменты выше, а дополняет их там, где нужно быстро пройтись по большой кодовой базе, найти экспортные методы, построить граф вызовов или оценить blast radius изменений.

### Когда вызывать rlm-tools-bsl
- Трассировка кода: «где вызывается процедура X?», «какие экспортные методы у модуля Y?», «от каких модулей зависит подсистема Z?»
- Оценка масштаба доработки до начала проектирования: сколько объектов затронуто, есть ли готовая реализация в конфигурации или расширении
- Ревью: найти всех callers изменённой функции, проверить согласованность сигнатур
- Поиск дубликатов перед рефакторингом или упрощением
- Разбор незнакомой конфигурации или расширения

### 5 tools сервера
- `rlm_projects` — реестр проектов (name → path). Actions: list/add/remove/rename/update. Мутации требуют пароль проекта
- `rlm_index` — управление SQLite-индексом (FTS5). Actions: build/update/info/drop. Обязателен перед первым анализом на больших (20K+ файлов) или медленных (HDD, сеть) базах — иначе glob/tree упрутся в таймауты. Мутации требуют `confirm=<пароль>`
- `rlm_start` — открыть сессию на каталоге. Параметры: `path` или `project`, `query` (человекочитаемое описание задачи), опциональный `effort` ∈ {low, medium, high, max}. Возвращает `session_id`
- `rlm_execute` — основной рабочий tool. Параметры: `session_id`, `code` (Python в read-only песочнице с BSL-хелперами). В контекст возвращается только `print()`. Таймаут 45 сек
- `rlm_end` — закрыть сессию. TTL: 10 мин idle / 30 мин active

### Ключевые хелперы внутри rlm_execute
- Навигация: `read_file`, `grep`, `grep_read`, `glob_files`, `tree`, `find_files`, `find_module`
- Анализ кода: `find_exports`, `extract_procedures`, `read_procedure`, `find_callers_context`
- Метаданные: `parse_object_xml`, парсеры форм/команд/атрибутов
- Расширения: специализированные хелперы CFE
- Sub-LLM: `llm_query`, `llm_query_batched` (квота по effort: 5/15/30/50 вызовов)

### Разовая настройка (один раз на проект)
- `rlm_projects` action=list — проверь, зарегистрирован ли проект. Если нет — `add` с путём и паролем
- `rlm_index` action=info — на большой/медленной базе (20K+ файлов, HDD, сеть) построй индекс через `rlm_index` action=build (требует `confirm=<пароль>`). На SSD с типовым проектом индекс не обязателен

### Обычный цикл работы
1. `rlm_start` — открой сессию с описанием задачи в `query` и `effort` по сложности
2. Один или несколько `rlm_execute` — детерминированные хелперы вместо LLM-запросов; дроби длинные обходы на несколько вызовов (таймаут 45 сек)
3. `rlm_end` — закрой сессию (или дай ей закрыться по TTL 10/30 мин)

Если по ходу работы видишь ошибку вида «проект не найден» или «индекс не актуален» — вернись к разовой настройке.

### Ограничения и предостережения
- Семантический поиск «по смыслу» не поддерживается — только детерминированные запросы по именам/файлам/шаблонам
- Таймаут `rlm_execute` — 45 сек. Длинные обходы дроби́ на несколько вызовов с промежуточными `print()`
- На каталоге расширения (Ext1) сервер запускать не нужно — открывай основную конфигурацию, расширения подхватываются автоматически
- Мутации (`rlm_index build/update/drop`, `rlm_projects add/remove/rename/update`) требуют пароль проекта — запрашивай его у пользователя, не генерируй
- На Windows Docker-вариант не рекомендуется (I/O через WSL2/Virtiofs медленнее в 5–10 раз)
